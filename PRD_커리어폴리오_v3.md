# PRD: AI 기반 커리어 포트폴리오 통합 플랫폼 (가칭: CareerFolio)

- 문서 버전: v3.0
- 작성일: 2026-07-09
- 대상 스택: Next.js (App Router) 풀스택 (설계 기준, 실제 구현 범위는 아래 참고)
- 상태: 컨셉 데모 (실 서비스 출시 아님)

> **⚠ 프로젝트 성격 안내**
> 본 프로젝트는 **실제 출시·실사용을 목표로 하지 않고, 제품 구상(컨셉)과 사용자 플로우를 보여주는 것을 목적**으로 합니다.
> - **AI/백엔드 API는 실제로 구현하지 않습니다.** Claude API, 파일 파싱, 인증, DB 등은 모두 **프론트엔드 목업(mock) 데이터·더미 응답**으로 대체하여 화면 흐름만 시연합니다.
> - **개발 기간은 1주일**로 잡으며, 목표는 "동작하는 서비스"가 아니라 "구상을 눈으로 확인할 수 있는 데모"입니다.
> - 아래 4장(기술 아키텍처)의 스택·API·데이터 모델은 **향후 실제 서비스화할 경우의 설계안**이며, 이번 1주 데모에서는 구현하지 않습니다.

## 1. 개요 (Overview)

### 1.1 제품 정의
사용자가 활동 자료(파일)를 업로드하면 AI가 이를 구조화하고, 대화형 인터뷰를 통해 성과·배운 점 등 인사이트를 추출한 뒤, 지원 직무에 맞춘 포트폴리오를 자동 생성·비교분석까지 제공하는 통합 커리어 관리 플랫폼 — **이라는 컨셉을, 실제 AI 연동 없이 목업 데이터로 시연하는 데모.**

원본 파일은 자체 서버·클라우드에 저장하지 않고, **사용자 본인의 Google Drive(Google 계정)** 에 저장하는 구조를 지향한다. 서비스는 파일 원본을 보관하지 않고 메타데이터(파일 ID·이름 등)만 참조하여, 사용자가 자기 데이터의 소유권을 그대로 유지할 수 있게 한다.

### 1.2 배경 및 문제 정의
| 문제 | 상세 |
|---|---|
| 시간·방법 부재 | 취준생 대다수가 포트폴리오 작성 시간이 없거나 방법을 모름 |
| 데이터 파편화 | 인턴·프로젝트·공모전 기록이 PDF, PPT, 이미지 등으로 분산 |
| 단순 나열 | "무엇을 했다" 수준에 그쳐 성과·인사이트가 드러나지 않음 |

### 1.3 타겟 사용자
- 20~30대 대학생 및 취업 준비생
- 경험은 있으나 스토리라인·성과 중심 표현에 서툰 사용자
- 파편화된 자료로 역량 어필에 어려움을 겪는 사용자

### 1.4 핵심 가치 제안 (Value Proposition)
"업로드만 하면, AI가 질문하고, 포트폴리오가 완성된다."
1. 자료 통합 → 2. 인사이트 추출 → 3. 포트폴리오 생성 → 4. 또래(동일 연령대) 비교 벤치마킹의 원스톱 파이프라인.

### 1.5 성공 지표 (참고용 – 실서비스화 시 목표)
> 아래 지표는 **실제 서비스로 출시할 경우의 목표치**입니다. 1주 컨셉 데모 단계에서는 이 지표를 측정하지 않으며, 대신 "4개 기능 화면이 전부 목업으로 매끄럽게 연결되는지"를 완성 기준으로 삼습니다.

| 지표 | 목표 (출시 후 3개월 가정) |
|---|---|
| 활동 카드 생성 완료율 (업로드 → 카드 생성) | ≥ 85% |
| 인터뷰 완주율 (활동당 전체 질문 답변 완료) | ≥ 60% |
| 포트폴리오 초안 생성 사용자 비율 | ≥ 40% |
| 초안 → 편집 후 내보내기(Export) 전환율 | ≥ 50% |
| 주간 재방문율 (WAU/MAU) | ≥ 35% |

## 2. 사용자 플로우 (User Flow)

```
회원가입/로그인 (목업 로그인 – 실 인증 없음)
   │
   ▼
[Feature 1] 활동 등록: 제목 + 기간(선택) + 파일 업로드
   │  (목업) 파일 파싱 → AI 정형화 → 활동 카드 생성
   ▼
[Feature 2] AI 인터뷰: 카드 기반 맞춤 질문 → 답변 → 피드백 루프
   │  (목업) Q&A 인사이트 저장 (중간 이탈 시 자동 저장 UI만 시연)
   ▼
[Feature 3] 포트폴리오 빌더: 직무 선택 → 초안 자동 생성 → 편집
   │  (목업) 사전 정의된 초안 템플릿 표시
   ▼
[Feature 4] 비교 분석: 동일 연령대(또래) 포트폴리오 vs 내 포트폴리오 갭 분석 → 개선 제안
   │  (목업) 사전 정의된 또래 그룹 리포트 표시
   ▼
내보내기 (PDF / 웹 공유 링크) — 목업 화면만 제공, 실제 파일 생성/링크 발급 X
```

## 3. 기능 요구사항 (Functional Requirements)

> 이하 각 Feature의 "처리 파이프라인/대화 로직/처리 과정"은 **실 서비스화 시의 설계안**이며, 1주 데모에서는 해당 로직을 실행하는 대신 **미리 준비한 목업 결과를 화면에 표시**하는 방식으로 구현합니다. (예: 실제 파일을 파싱하지 않고, 업로드 즉시 정해진 목업 활동 카드를 보여줌)

### Feature 1. 활동별 파일 업로드 및 커리어 DB 자동 정형화

**목적:** 흩어진 활동 자료를 한 곳에 모으고 AI가 구조화된 데이터로 변환.

| 항목 | 명세 |
|---|---|
| 입력 | 활동명(필수), 활동 기간(선택), 파일 1개 이상 (다중 업로드) |
| 지원 형식 | PDF, JPG/PNG, XLSX, PPTX, DOCX, TXT |
| 제약 | 파일당 최대 20MB, 활동 1건당 최대 5개 |
| 출력 | 활동 카드: 제목, 요약, 기간, 주요 역할, 키워드 태그 |

**처리 파이프라인 (설계안 · 데모에서는 목업으로 대체):**
1. 최초 이용 시 Google OAuth 동의(Drive 파일 쓰기 권한, `drive.file` scope)를 받아 사용자 Google 계정과 연동
2. 클라이언트에서 Google Drive API로 사용자 본인 드라이브 내 "CareerFolio" 전용 폴더에 파일 직접 업로드 (자체 서버에는 파일 원본 미보관)
3. 업로드 완료 → 반환된 Google Drive `fileId`를 기준으로 파싱 잡(job) 큐 등록
4. 텍스트 추출: Drive에서 파일을 임시로 읽어와 PDF(`pdf-parse`), DOCX(`mammoth`), XLSX(`sheetjs`), PPTX(zip+XML 파싱), 이미지(Claude Vision 멀티모달 입력) 후 원본은 서버에 남기지 않음
5. 추출 텍스트를 Claude API에 전달 → JSON 스키마로 활동 개요·역할·키워드 구조화
6. 활동 카드 DB에는 구조화 결과와 Drive `fileId`만 저장 → 클라이언트에 실시간 상태 갱신(폴링 or SSE)

**데모 구현 방식:** "Google 계정으로 연동" 버튼을 누르면 실제 OAuth 없이 목업 동의 화면만 보여준 뒤, 파일을 업로드하면 실제 Drive 저장·파싱 없이 2~3초 로딩 애니메이션 후 파일 종류에 따라 미리 준비한 목업 활동 카드(제목/요약/키워드) 3~4종 중 하나를 매칭해 보여준다.

**예외 처리 (설계안):**
- 텍스트 추출 실패(저화질 스캔 등) → 카드 상태 `NEEDS_MANUAL_INPUT`로 표시, 수동 보완 입력 폼 제공
- 지원하지 않는 형식/용량 초과 → 업로드 전 클라이언트 검증 + 서버 이중 검증
- AI 파싱 실패 → 3회 재시도 후 수동 입력 유도

**Acceptance Criteria (데모 기준):**
- [ ] 파일 업로드 시 진행률 UI가 표시된다
- [ ] 업로드 후 3초 이내 목업 활동 카드가 화면에 나타난다
- [ ] 특정 예외 케이스(예: 이미지 파일) 선택 시 "수동 입력 필요" 상태 UI를 보여줄 수 있다

### Feature 2. AI 기반 활동 인사이트 인터뷰

**목적:** 대화형 질의응답으로 성과와 배운 점을 구체적인 언어로 추출.

| 항목 | 명세 |
|---|---|
| 입력 | Feature 1의 활동 카드 (개요, 역할, 키워드) |
| 질문 카테고리 | ① 문제 상황·극복 과정 ② 정량/정성 성과 ③ 배운 점·아쉬운 점 |
| 출력 | Q&A 데이터 (질문, 답변, AI 요약 핵심 한 줄) |

**대화 로직 (설계안):**
- 카드의 역할/키워드 기반으로 카테고리별 맞춤 질문을 순차 생성 (활동당 기본 5~8문항)
- 답변 수신 → AI가 구체성 평가:
  - 충분 → 핵심 한 줄 요약 저장 후 다음 질문
  - 부족 → 재질문 (최대 2회, 이후 다음 질문으로 강제 진행하여 피로 방지)
  - "모르겠다"/회피 → 유사 사례 예시 제시 후 구체화 유도
- (설계안) 스트리밍 응답(Claude API streaming)으로 대화 체감 속도 확보

**데모 구현 방식:** 실제 AI 호출 없이, 활동 카드 종류별로 미리 작성한 질문 스크립트(5~7문항)를 순서대로 노출하고, 사용자가 답변을 입력하면 타이핑 애니메이션과 함께 미리 작성된 AI 피드백 문구를 보여준다.

**저장 방식 (설계안):**
- 답변 제출 즉시 DB 저장 (자동 저장). 세션 이탈 후 재진입 시 마지막 질문부터 이어서 진행
- 인터뷰 상태: `NOT_STARTED` / `IN_PROGRESS` / `COMPLETED`

**Acceptance Criteria (데모 기준):**
- [ ] 답변 입력 후 미리 정의된 AI 피드백 문구가 자연스럽게 이어진다
- [ ] "모르겠다" 입력 시 예시 포함 재질문 스크립트가 노출된다
- [ ] 마지막 문항 완료 시 "인터뷰 완료" 상태로 전환되고 활동 카드에 표시된다

### Feature 3. 데이터 연동형 맞춤 포트폴리오 자동 빌더

**목적:** 활동 데이터 + 인터뷰 인사이트 + 지원 직무를 종합해 포트폴리오 초안 자동 생성.

| 항목 | 명세 |
|---|---|
| 입력 | 전체 활동 카드 + Q&A 데이터 + 지원 직무 선택 (마케팅/개발/데이터분석/기획/디자인 등) |
| 처리 | ① 직무 관련성 기반 활동 우선순위화 ② 직무 스타일 문체·구성 재구성 ③ 초안 생성 |
| 출력 | 제목, 자기소개, 활동별 상세(스토리라인), 핵심 성과 요약 |

**편집 기능:**
- 블록 기반 에디터: 문장 단위 수정, 활동 순서 드래그 변경, 특정 활동 제외 토글
- AI 부분 재작성: 문단 선택 → "더 간결하게 / 성과 강조 / 직무 키워드 반영" 리라이트 (설계안 · 데모에서는 클릭 시 미리 준비한 재작성 예문으로 치환)
- 버전 이력: 초안 생성 시점 스냅샷 저장, 되돌리기 지원

**데모 구현 방식:** 직무를 선택하면 실제 생성 없이 2초 내외 로딩 후, 선택한 직무에 맞춰 미리 작성해 둔 포트폴리오 초안 템플릿을 채워 넣어 보여준다. 블록 편집·드래그 순서 변경 등 UI 조작은 실제 동작하되, 결과는 클라이언트 상태에만 반영(저장 X).

**예외 처리 (설계안):**
- 인터뷰 미완료 활동 → 초안에서 제외하거나 "정보 보완 필요" 경고 배지 표시 (사용자 선택)
- 활동 데이터 부족(카드 1개 미만) → 생성 차단 + 활동 등록 유도

**Acceptance Criteria (데모 기준):**
- [ ] 직무 선택 후 목업 초안이 화면에 표시된다
- [ ] 미완료 인터뷰 활동에 경고 배지가 표시되고, 클릭 시 인터뷰 화면으로 이동한다
- [ ] 블록 순서 변경·문단 편집 UI가 클라이언트 상에서 정상 동작한다

### Feature 4. 또래(동일 연령대) 포트폴리오 비교 분석

**목적:** 나와 비슷한 나이대·직무를 지원하는 또래 사용자들의 포트폴리오(익명화)와 비교해 벤치마킹 인사이트 제공.

| 항목 | 명세 |
|---|---|
| 입력 | 사용자 출생연도/나이 + 지원 직무 + 사용자 포트폴리오 초안 |
| 데이터 소스 | 플랫폼 내 동의 기반 익명화 또래 포트폴리오 DB (동일·인접 연령대 사용자, 예: ±2세) |
| 출력 | ① 또래 그룹 내 공통 패턴 요약 ② 갭 분석 리포트 ③ 개선 제안 리스트 |

**처리 과정 (설계안):**
1. 사용자와 동일·인접 연령대(예: 만 나이 ±2세) & 동일·유사 직무군 사용자 포트폴리오 필터링 (연령 구간 + 직무 임베딩 유사도 기반)
2. AI가 또래 그룹 내 공통 패턴 추출: 강조 활동 유형, 성과 표현 방식(정량 지표 위치 등), 구성 순서
3. 사용자 포트폴리오와 또래 패턴 비교 → 갭 스코어링
4. 차이점·개선 포인트 시각화 (레이더 차트, 항목별 체크리스트)

**데모 구현 방식:** 실제 또래 DB·임베딩 검색 없이, 연령대·직무 조합별로 미리 만들어 둔 목업 리포트(갭 스코어, 레이더 차트 값, 개선 제안 3개)를 선택 조건에 맞춰 보여준다. "표본 부족" 폴백 화면도 특정 조합(예: 희귀한 나이대×직무 조합)을 선택했을 때 노출되도록 고정 시나리오로 구성한다.

**예외 처리 (설계안):**
- 해당 연령대·직무 조합 표본 부족(예: n < 10) → 연령대 범위 확장(±2세 → ±4세) 또는 유사 직무군 확장 비교, 그래도 부족 시 "데이터 부족" 안내 + 일반 가이드 제공

**개인정보 보호 (설계안 – 실 데이터 미보유):**
- 비교 DB의 모든 사례는 이름·소속·사진 등 식별 정보 제거 후 저장, 나이는 연령대 구간으로만 저장(정확한 생년월일 비저장)
- 사용자 포트폴리오는 명시적 opt-in 동의 없이는 비교 DB에 축적되지 않음
- 동의 철회 시 30일 이내 DB에서 삭제

**Acceptance Criteria (데모 기준):**
- [ ] 나이대·직무를 선택하면 목업 갭 분석 리포트(레이더 차트 + 체크리스트)가 표시된다
- [ ] 특정 조합 선택 시 "또래 데이터 부족" 폴백 화면이 노출된다
- [ ] 개선 제안이 최소 3개 이상 목업 데이터로 노출된다

## 4. 기술 아키텍처 — 설계안 (실서비스화 시 참고, 1주 데모에서는 미구현)

> 이 장은 **"나중에 진짜 서비스로 만든다면 이렇게 만들겠다"는 설계 문서**입니다. 1주 데모는 Next.js 프론트엔드 + 클라이언트 목업 데이터(JSON)만으로 구성하며, 아래의 Google Drive 연동/DB/Claude API/인증 등은 **실제로 연동하지 않습니다.**

### 4.1 기술 스택 (설계안)

| 레이어 | 선택 | 비고 | 1주 데모 적용 여부 |
|---|---|---|---|
| 프레임워크 | Next.js 15 (App Router) + TypeScript | 풀스택 (RSC + Server Actions + Route Handlers) | ✅ 사용 (프론트만) |
| UI | Tailwind CSS + shadcn/ui | 블록 에디터는 Tiptap | ✅ 사용 |
| 상태 관리 | TanStack Query + Zustand | 서버 상태 / 클라이언트 UI 상태 분리 | △ 목업 데이터 fetch 흉내에만 사용 |
| 인증 | Auth.js (NextAuth v5) | 이메일 + 카카오/구글 OAuth (구글 로그인 시 Drive `drive.file` scope 함께 요청) | ❌ 미구현 (목업 로그인 화면만) |
| DB | PostgreSQL + Prisma ORM | 배포: Supabase 또는 Neon | ❌ 미구현 (정적 JSON으로 대체) |
| 벡터 검색 | pgvector | 연령대·직무 유사도 기반 또래 사례 검색 | ❌ 미구현 |

| 모니터링 | Sentry + Vercel Analytics | | ❌ 데모 단계 불필요 |

### 4.4 데이터 모델 (설계안 – 실서비스화 시 Prisma 스키마, 데모에서는 아래 형태의 TS 목업 객체로 대체)

```prisma
model User {
  id                String      @id @default(cuid())
  email             String      @unique
  name              String?
  birthYear         Int?        // Feature 4 또래 비교용 출생연도
  targetJob         String?     // 지원 직무
  consentToDb       Boolean     @default(false) // Feature 4 또래 비교 DB 축적 동의
  googleDriveLinked Boolean     @default(false) // 구글 드라이브 연동 여부
  googleRefreshToken String?    @db.Text // OAuth refresh token (암호화 저장)
  activities        Activity[]
  portfolios        Portfolio[]
}

model Activity {
  id          String        @id @default(cuid())
  userId      String
  title       String
  periodStart DateTime?
  periodEnd   DateTime?
  summary     String?       // AI 생성 요약
  role        String?       // 주요 역할
  keywords    String[]      // 키워드 태그
  status      ActivityStatus @default(PROCESSING)

  interview   Interview?
}

enum ActivityStatus { PROCESSING  READY  NEEDS_MANUAL_INPUT  FAILED }

model ActivityFile {
  id                String   @id @default(cuid())

  googleDriveFileId  String   // 사용자 Drive 내 "CareerFolio" 폴더의 파일 ID (원본은 서버에 미보관)

  mimeType          String
  sizeBytes         Int
  extractedText     String?  @db.Text // 파싱 시 임시로 읽어 추출한 텍스트만 저장
  parseStatus       String    // PENDING / DONE / FAILED
}

model Interview {
  id         String          @id @default(cuid())

}

  id          String  @id @default(cuid())
  interviewId String
  category    String  // PROBLEM / RESULT / LESSON
  question    String  @db.Text
  answer      String? @db.Text
  keyInsight  String? // AI 핵심 한 줄 요약

}

  id        String   @id @default(cuid())
  userId    String
  targetJob String
  content   Json     // 블록 기반 구조 (제목/소개/활동섹션/성과요약)
  version   Int      @default(1)

}

model AnalysisReport {
  id           String   @id @default(cuid())
  portfolioId  String
  patterns     Json     // 또래 그룹(연령대x직무) 공통 패턴
  gaps         Json     // 갭 분석 결과
  suggestions  Json     // 개선 제안 리스트
  createdAt    DateTime @default(now())
}

model PeerCase {             // 익명화된 또래 포트폴리오 사례
  id         String                    @id @default(cuid())
  ageGroup   Int                       // 연령대 구간 (예: 만 나이 기준 그룹 값)
  jobFamily  String
  content    Json                      // PII 제거 완료 데이터
  embedding  Unsupported("vector")?    // pgvector
  source     String                    // INTERNAL_OPTIN
}
```

### 4.5 핵심 API 명세 (설계안 — **1주 데모에서는 실제로 구현하지 않음**)

> 아래 엔드포인트는 향후 실서비스화 시의 설계 참고용입니다. 데모에서는 이 API들을 호출하는 대신, 동일한 시그니처의 함수가 `src/mock/*.ts`의 정적 데이터를 즉시(또는 setTimeout으로 지연 흉내) 반환하도록 대체합니다.

| Method | Endpoint (설계안) | 설명 | 데모 대체 방식 |
|---|---|---|---|
| POST | `/api/google/oauth/callback` | Google OAuth 인증 콜백, refresh token 저장 | 미호출, 목업 "연동 완료" 화면만 표시 |
| POST | `/api/activities` | 활동 생성 + Google Drive 업로드 결과(fileId) 저장 + 파싱 잡 등록 | mock 활동 카드 즉시 반환 |
| GET | `/api/activities/:id` | 카드 상태·데이터 조회 (폴링용) | mock 데이터 조회 (지연 없음) |
| POST | `/api/interview/stream` | 인터뷰 질문/피드백 SSE 스트리밍 | mock 스크립트를 타이핑 효과로 순차 표시 |
| PATCH | `/api/interview/:id/answer` | 답변 저장 (자동 저장) | 클라이언트 상태(Zustand)에만 저장 |
| POST | `/api/portfolio/generate` | 직무 기반 초안 생성 | 직무별 mock 템플릿 반환 |
| PATCH | `/api/portfolio/:id` | 편집 내용 저장 (블록 JSON) | 클라이언트 상태에만 저장 |
| POST | `/api/analysis` | 비교 분석 실행 → 리포트 반환 | 나이대·직무 조합별 mock 리포트 반환 |

### 4.6 AI 프롬프트 설계 원칙 (설계안 — 실서비스화 시 사용할 프롬프트 방향성, 데모에서는 미사용)
- **구조화(Feature 1):** JSON 스키마 강제 + 필드별 누락 시 `null` 허용, 환각 방지를 위해 "파일에 없는 정보는 추측하지 말 것" 명시
- **인터뷰(Feature 2):** STAR 프레임워크(Situation-Task-Action-Result) 기반 질문 생성, 답변 구체성 평가 루브릭 포함
- **생성(Feature 3):** 직무별 스타일 가이드(어휘·구성 템플릿)를 시스템 프롬프트에 주입, 사용자의 실제 답변만 근거로 사용
- **분석(Feature 4):** 또래(동일 연령대) 사례 원문 노출 금지 — 패턴 요약만 출력하여 타인 포트폴리오 유출 방지

| 확장성 | 파싱 워커 수평 확장 가능 구조, DB 커넥션 풀링(PgBouncer) |

**1주 데모 기준 실질 요구사항:** 데스크톱 기준 주요 브라우저(Chrome)에서 4개 Feature 화면이 끊김 없이 전환되고, 목업 데이터가 화면 깨짐 없이 표시되면 충분함.
